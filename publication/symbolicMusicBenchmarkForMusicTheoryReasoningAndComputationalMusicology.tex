\begin{abstract}
We present a reproducible benchmark for evaluating music-theory reasoning in large language models (LLMs) under symbolic input-output contracts. The benchmark defines eight zero-shot tasks in two families: label classification (interval identification, chord identification, harmonic-function classification, and voice-leading violation detection) and note-sequence transformation (transposition, melodic inversion, retrograde, and rhythm scaling). Cases are generated by deterministic rule-based procedures, exported into three tokenizer views (\texttt{note\_level}, \texttt{midilike}, and \texttt{remilike}), and evaluated with automatic verifiers. The benchmark scope is intentionally constrained to ensure stable evaluation: harmonic function is defined in major-mode context, voice-leading uses a two-voice setting, and chord vocabulary is limited to a compact tonal subset. We standardize prompting with strict system-level output constraints and instance-only user content, and support both local and API model backends through one execution suite. This benchmark is designed as an engineering artifact for controlled comparison and as a computational musicology tool for analyzing the gap between symbolic correctness and musical reasoning.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Recent LLM-based systems can generate plausible symbolic music sequences, but generation quality alone does not establish music-theory competence. A model may produce fluent token sequences while failing on core analytical operations such as interval naming, harmonic-function assignment, or deterministic melodic transformation.

This paper targets that gap with a benchmark centered on \emph{music-theory reasoning under strict symbolic contracts}. Instead of scoring open-ended musicality, we ask whether a model can produce verifiable outputs for tasks that correspond to standard analytical or transformational concepts.

Our design priorities are: (1) deterministic ground truth, (2) tokenizer-agnostic case semantics, (3) strict machine-checkable outputs, and (4) reproducible end-to-end execution. Compared with broad multi-task LLM-in-music evaluations \cite{zhou2024llmsmusicreason}, we focus on narrower but cleaner tasks where scoring is fully automated.

\textbf{Contributions.}
(1) We define an eight-task symbolic benchmark that separates label prediction from note-sequence transformation.
(2) We provide a deterministic generation and verification pipeline with synchronized tokenizer views.
(3) We standardize prompt construction into a reusable system/user contract for local and API runners.
(4) We report reproducibility checks and pilot observations, and document explicit scope limits for future expansion.

\section{Related Work}
\label{sec:related_work}

Symbolic music modeling has progressed through event-based tokenization and pretraining approaches, including REMI and MusicBERT \cite{huang2020pop,huang2021musicbert}. However, common evaluations emphasize continuation quality or downstream prediction rather than explicit theory-grounded reasoning.

Recent studies on LLMs for music understanding and generation evaluate broader capability sets and multi-step prompting behavior \cite{zhou2024llmsmusicreason}. Our benchmark is complementary: we reduce task breadth to improve scoring determinism and reproducibility. More generally, the benchmark follows principles from holistic LLM evaluation work emphasizing transparent protocol design and robust reporting \cite{liang2023holistic}.

\section{Methodology}
\label{sec:design}

\subsection{Task Taxonomy}

The benchmark defines eight tasks grouped by output type.

\textbf{Label classification:}
\begin{itemize}
    \item Task 1: Interval Identification
    \item Task 2: Chord Identification
    \item Task 3: Harmonic Function Classification
    \item Task 4: Voice-Leading Violation Detection
\end{itemize}

\textbf{Note-sequence transformation:}
\begin{itemize}
    \item Task 5: Transposition
    \item Task 6: Melodic Inversion
    \item Task 7: Retrograde
    \item Task 8: Rhythm Scale
\end{itemize}

For label tasks, each case has one canonical target label. For sequence tasks, each case has one target event sequence.

\subsection{Deterministic Case Generation}

All cases are generated by rule-based scripts with fixed seeds. The generator creates canonical note-level payloads and deterministic targets, including event timing and duration fields for sequence tasks.

Examples of deterministic operators include semitone transposition, inversion around pivot, order reversal, and duration scaling. For rhythm scaling, duration pools are chosen to keep outputs aligned with the benchmark grid.

\subsection{Tokenizer Views}

All cases are generated in a canonical event schema with fields t (start time), p (MIDI pitch), d (duration), and v (velocity). The export step then serializes the same payload into three synchronized tokenizer views:
\begin{itemize}
    \item note\_level: event blocks separated by \texttt{|}, e.g., \texttt{t=0.5 d=0.25 notes=[G4] v=80}
    \item \texttt{midilike}: repeating quintuplets, e.g., \texttt{TIME\_0.5 NOTE\_ON\_67 VEL\_80 DUR\_0.25 NOTE\_OFF\_67}
    \item \texttt{remilike}: repeating quadruplets, e.g., \texttt{POS\_0.5 PITCH\_67 VEL\_80 DUR\_0.25}
\end{itemize}

For pitch-set inputs in label tasks, serialization is also tokenizer-specific: note\_level uses bracketed note-name sets (e.g., \texttt{[C4,E4,G4]}), \texttt{midilike} uses \texttt{CHORD\_NOTE\_60 CHORD\_NOTE\_64 CHORD\_NOTE\_67}, and \texttt{remilike} uses pitch-class tokens (e.g., \texttt{PitchClass\_0 PitchClass\_4 PitchClass\_7}).

During view export, input/target strings and prompt text are regenerated for each tokenizer, while case identity and underlying musical payload remain fixed. Validation then applies two consistency checks: identical case ordering across tokenizer files, and exact match between decoded sequence targets and the canonical event-level targets.

\subsection{Prompt Design and Composition}

Each benchmark item is prompted in a two-part format: a system instruction block and a user content block.

The \textbf{system block} is shared across items of the same task/tokenizer setting and defines hard constraints: required output type (single label or transformed note sequence), no extra text, and tokenizer-specific output grammar.

The \textbf{user block} is instance-specific and follows a fixed composition order:
\begin{itemize}
    \item one task description line;
    \item condition/control lines when applicable (e.g., key shift, inversion pivot, rhythm factor, timing context);
    \item serialized musical input line(s), rendered in one of three tokenizer views (\texttt{note\_level}, \texttt{midilike}, \texttt{remilike}).
\end{itemize}

For label tasks (Task 1--4), the expected output is exactly one label token. For sequence tasks (Task 5--8), the expected output is exactly one transformed token sequence in the requested tokenizer format.

We use deterministic line ordering for user content so that prompts are stable across runs and directly auditable in exported case files. We do not use chain-of-thought exemplars, in-context QA examples, or multi-round decomposition in the benchmark protocol.

Two prompt modes are supported:
\begin{itemize}
    \item \texttt{light}: concise instructions with minimal formatting constraints.
    \item \texttt{agent\_like}: strict output-only rules, explicit no-extra-text constraints, tokenizer grammar constraints, and task-specific operation rules.
\end{itemize}

\section{Experimental Protocol}
\label{sec:protocol}

\subsection{Execution Pipeline}

The protocol is designed to isolate model behavior from data noise. We first generate canonical symbolic cases, then export tokenizer-specific views, and finally execute model inference on a unified case schema. Predictions are reattached to their source cases for item-level analysis. A regeneration policy (\texttt{always}/\texttt{auto}/\texttt{never}) controls when benchmark inputs are rebuilt, so repeated runs can be either strictly fresh or strictly comparable depending on the experiment goal.

\subsection{Model Backends}

The same benchmark cases are executed across local runtimes (Ollama) and API runtimes (OpenAI-compatible and Gemini-compatible endpoints). We use one provider-normalized interface and explicit provider-prefixed model identifiers, which keeps model routing transparent while holding the case format fixed. Each case is sent as an independent request, so no cross-item conversational state can influence scores.

\subsection{Metrics}

The primary score is binary hit/miss for all tasks. A prediction is a \emph{hit} only when it exactly matches the ground truth (label string for classification tasks, full event sequence for transformation tasks); otherwise it is a \emph{miss}. No partial credit is used in the main benchmark score.

This strict definition keeps task comparisons interpretable across tokenizers and model families. Prediction files are post-processed into case-level annotations with match status; optional diagnostics (e.g., text or note-level error rates) are retained for debugging but are not part of the primary score.

\section{Results and Analysis}
\label{sec:results}

\subsection{Pipeline Sanity Checks}

Oracle evaluations achieve perfect scores by construction, confirming that generation, tokenizer rendering, and evaluators are internally consistent under gold predictions.

\subsection{Pilot Model Behavior}

Repository pilot runs show heterogeneous behavior across tasks and tokenizers. Label tasks are generally more stable than sequence transformation tasks, while strict token formatting can dominate failure modes in sequence outputs. This pattern is consistent with the benchmark's design goal: separate theory errors from format-compliance errors.

\subsection{Observed Failure Modes}

Case-level annotations show recurring miss categories:
\begin{itemize}
    \item invalid output format (empty output, malformed tokens),
    \item operation mismatch (wrong inversion, retrograde, or duration scaling),
    \item label/sequence mismatch against the exact ground truth.
\end{itemize}

These failures are inspectable through stored per-case artifacts, enabling direct debugging of prompt or decoding strategies.

\section{Discussion}
\label{sec:discussion}

\subsection{What This Benchmark Measures}

The benchmark measures \emph{structured symbolic reasoning under constraints}, not open-ended compositional quality. It is useful for controlled comparison of model behavior on formal theory operations.

\subsection{Current Limits}

The release is intentionally narrow: major-mode harmonic function, reduced chord vocabulary, and two-voice voice-leading. It does not yet evaluate richer tonal contexts, counterpoint regimes, or stylistic interpretation.

\subsection{Next Steps}

Priority extensions include expanded harmonic contexts, more complex voice-leading settings, constrained decoding at token level, and larger-scale comparative runs with stable statistical reporting.

\section{Conclusion}
\label{sec:conclusion}

We introduced a reproducible symbolic benchmark for music-theory reasoning in LLMs. The framework combines deterministic case generation, synchronized tokenizer views, strict prompt contracts, and automatic scoring for both label and sequence tasks. By design, the benchmark emphasizes reliability and inspectability over maximal theory breadth, providing a concrete base for future computational musicology evaluation.
